## ★ 重大: 設計の前提が崩れている箇所

### 1. Web Speech API はフォールバックとして成立しない

roadmap.md は「精度が出ない場合は Web Speech API へ切替」としているが、検証の結果:

- **iOS のスタンドアロン PWA(ホーム画面追加)では SpeechRecognition が動作しない**(APIは存在するが結果が返らない既知問題)
- **サーバー処理のためオフラインでは動かない**。「機内モードでも全機能が動作」の非機能要件と矛盾

つまり「Whisper がダメなら Web Speech API」という保険は、PWA 運用でもオフライン要件でも破綻している。
**対応案**: フェーズ1 の比較対象としては残してよい(Safari タブでの精度ベースライン測定用)が、
design.md から「代替案」としての位置づけを外し、真の代替案を再定義する
(例: 候補ワードのタップ入力 UI を磨く / 認識は Wi-Fi のある自宅で後処理する「あとで文字起こし」モード等)。

### 2. kotoba-whisper のサイズ見積りが大幅に誤っている

design.md §6 は「kotoba-whisper 系 約100〜150MB」とするが、実際は
**onnx-community/kotoba-whisper-v2.2-ONNX はエンコーダ単体の q4f16 で 370MB**。デコーダを足すとさらに大きい。
distil-large-v3 ベースなので whisper-small(190MB)より重く、iPhone Safari のメモリでは実行困難の可能性が高い。

**対応案**: モデル候補表を修正。現実的な候補は whisper-tiny(~40MB)/ whisper-base(~60MB)/
whisper-small(~190MB, メモリ次第)。kotoba-whisper は「実機で試すが期待薄」と明記するか候補から外す。

### 3. ささやき声 × Whisper の幻覚(ハルシネーション)対策が設計にない

Whisper は無音・小音量の日本語音声で「ご視聴ありがとうございました」等を幻覚出力することで有名
(学習データ由来の既知問題)。ささやき声はまさにこの発生条件に近い。
パーサーが種目不明として弾くので誤保存はされにくいが、頻発すると使い物にならない。

**対応案(design.md §6 に追記)**:
- 録音波形の RMS(音量)チェックで無音に近い録音は推論に回さない
- 既知幻覚フレーズのブラックリストでパーサー前に除去
- 将来: Silero VAD(これも ONNX でローカル実行可、コストゼロ制約に適合)

### 4. マイク権限プロンプトが毎回出る可能性(iOS PWA の既知問題)

iOS のホーム画面 Web アプリでは getUserMedia の権限が持続せず、
セッションごとに再プロンプトされる報告が多数ある(WebKit の既知バグ)。
「ジムでセット間 30 秒の間にサッと記録」という核心 UX が、毎回の許可タップで崩れるリスク。

**対応案**: フェーズ1 の実機検証項目に「アプリ再起動・画面ロック復帰後の権限プロンプト頻度」を明記。
ストリームを画面遷移で解放せず使い回す実装方針もあわせて記載。

## 高: データ損失・誤記録につながる穴

### 5. Safari タブ運用と 7 日 ITP 消去の衝突

Safari は 7 日間サイトを訪れず使い続けると IndexedDB / Cache Storage を消去する。
ホーム画面追加なら免除されるが、design.md §10 の「(PWA でマイクが)ダメなら Safari タブ運用にフォールバック」
を採った場合、**筋トレを 1 週間サボる+Safari は使う → 全記録とモデル(60MB)が消える**。
現状の設計はこの相互作用に触れていない。

**対応案**: `navigator.storage.persist()` の呼び出しを初期化に追加、
ホーム画面追加を強く促すオンボーディング、エクスポート促しバナー(実装済み計画)の 3 点セットを design に明記。

### 6. パーサーのバグ: 「62点5キロ」が weightKg=5 になる

normalize は**漢数字に続く「点」しか変換しない**ため、Whisper が出しがちな
「62点5キロ」(算用数字+点)が未変換のまま残り、WEIGHT_RE が末尾の「5キロ」にマッチして
**誤った重量 5kg を黙って返す**。null ではなく誤値なのが危険(「推測で埋めない」原則の実質的な違反)。

**対応案**: normalize に `数字+点+数字 → 小数点` の変換を追加し、テストケース
`parse("ベンチプレス 62点5キロ 10回")` → 62.5 を追加。同様に「62てん5」も検討。

### 7. プランク等の時間ベース種目がデータモデルで表現できない

exercises.ts のシードに「プランク」があるが、sets は reps 必須で秒数フィールドがない。
「プランク 60秒」は記録不能(60秒 → 重量にも回数にもマッチせず保存不可)。

**対応案**: MVP では (a) プランクをシードから外し design で明示的にスコープ外とする、
または (b) `sets.durationSec` を今のうちにスキーマへ追加(スキーマ確定前なら移行コストゼロ)。

### 8. 前回参照(F5)が「挿入順」で最後のセットを返す

`lastSetOfExercise` は主キー順の `.last()` のため、過去のトレーニングを後から手入力すると
それが「前回」として表示される。**対応案**: `[exerciseId+performedAt]` の複合インデックスを
version(1) のうちに追加(リリース前ならマイグレーション不要)。

## 中: 品質・堅牢性

### 9. 部分一致・あいまい一致の曖昧性を検出しない

「レッグ」はレッグプレス/レッグエクステンション/レッグカール/レッグレイズに部分一致しうるが、
現実装は辞書順で最初の 1 件を silent に採用する。fuzzy も最良 1 件のみで次点との距離差を見ない。
**対応案**: 複数ヒット時は confidence を "ambiguous" 扱いにして確認 UI で候補から選ばせる。

### 10. IndexedDB 初期化失敗で白画面

main.tsx は `seedIfEmpty()` が reject すると何も描画しない。プライベートブラウズや容量逼迫で発生しうる。
**対応案**: try/catch でエラー画面(手動入力のみモード or 説明文)を出す。

### 11. 単位なし数値の発話に非対応

Whisper は「ベンチプレス 60 10回」のように単位を落とすことがある。現状 60 は捨てられ weightKg=null。
**対応案**: MVP 後でよいが、「数値が 2 つ残ったら 重量→回数 の順とみなす」ヒューリスティックを検討事項として design に記載。

### 12. WebGPU の活用が設計に反映されていない(好材料)

iOS 26 で Safari が WebGPU に正式対応済み(2026 年現在)。transformers.js は `device: "webgpu"` 指定のみで
利用でき、WASM 比で大幅な高速化が見込める。「発話終了から 10 秒以内」の達成可能性を上げる材料。
**対応案**: フェーズ1 の計測マトリクスを「モデル × (WASM / WebGPU)」にし、WebGPU 不可端末への WASM フォールバックを設計に明記。

### 13. iOS はマニフェストの icons を無視する

ホーム画面アイコンには `<link rel="apple-touch-icon">` が必須(index.html に現状なし)。
アイコン整備はフェーズ5 予定だが、フェーズ1 で「ホーム画面追加」を検証する以上、
**仮アイコン+apple-touch-icon はフェーズ1 に前倒し**すべき(検証対象の一部のため)。

### 14. インポート機能がない(エクスポートの片翼)

F7 はエクスポートのみで、機種変更・データ消失後に**戻す手段がない**。バックアップとしての価値が半減。
**対応案**: JSON インポートを MVP に含める(エクスポートと同じ画面に置く。実装は小さい)。

### 15. 録音音声のデコード方針が未記載

iOS Safari の MediaRecorder は audio/mp4(AAC)を出力する。Whisper 入力には
`decodeAudioData` → 16kHz モノラル Float32Array へのリサンプリングが必要。
フェーズ1 でハマりやすいポイントなので design §6 に処理系列を 1 行書いておく。

## 低: 記録のみ

- HistoryScreen が全セットを一括ロード(MVP 規模では問題なし。数年分溜まったらページング)
- 種目追加 UI(フェーズ4)では `&name` / `&alias` の unique 制約違反時のエラー処理が必要
- GitHub Actions ワークフローが未作成(roadmap フェーズ1 の未着手項目。git リポジトリ自体は初期化済み)
- 参考: 英語の静穏環境データでは whispered speech の WER 劣化は large-v2 で 16.7%→18.8% と小さいという報告があるが、base 級・日本語・ジム騒音では大きく悪化しうる。フェーズ1 検証の優先は正しい

## 良い点(維持すべき設計判断)

パーサー純粋関数+テスト 22 件、DB アクセスの窓口一元化、performedAt をローカル時刻で持つ判断(UTC の前日問題を回避)、
リスク先行のフェーズ順(技術検証を UI より先に)、コストゼロ検証表 — いずれも妥当で、実装は設計に忠実。
