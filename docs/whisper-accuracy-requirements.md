# ささやき声 認識精度改善 — 要件定義

作成: 2026-07-20 / 状態: **提案(未着手)** / 一次情報源: design.md §6 と本書

ジムでのささやき声入力に対する Whisper の認識精度が低い問題への対策を、
実装要件として整理する。**コストゼロ制約(CLAUDE.md)は全要件に適用される**
(クラウド STT・有料 API・課金の可能性があるサービスは選択肢に入れない)。

## 1. 現状整理(2026-07-20 コード調査)

| 項目 | 状態 |
|---|---|
| 録音設定 | `echoCancellation: true` / `noiseSuppression: true`(recorder.ts)。**`autoGainControl` は未指定** |
| 音量ゲート(RMS) | design.md §6 に記載あり、**未実装** |
| 幻覚ブラックリスト | design.md §6 に記載あり、**未実装** |
| 音量の可視化 | なし(録音中に声が届いているか分からない) |
| モデル | whisper-base / q4 / WASM が既定(modelConfig.ts)。VerifyScreen で切替比較可 |
| 推論オプション | `language: "japanese"`, `task: "transcribe"` のみ(transcriber.ts) |
| パーサー | 完全一致→別名→部分一致→編集距離(閾値: 文字数の 1/3)の段階マッチ実装済み(parse.ts) |

## 2. 課題の構造(なぜささやき声で外れるか)

1. **声帯振動がない** — 有声/無声の区別が消え、清音・濁音・半濁音の混同
   (バ/パ/ハ、ガ/カ 等)が起きやすい。「ベンチ」→「ペンチ」型の誤り
2. **信号レベルが低い** — S/N 比が悪化し、ジムの環境音(音楽・金属音)に埋もれる
3. **ブラウザのノイズ抑制が逆効果の可能性** — noiseSuppression は有声音声向けに
   調整されており、ささやき声を「ノイズ」として削る恐れがある(未検証・要 A/B)
4. **幻覚の発生条件に合致** — 無音に近い入力で「ご視聴ありがとうございました」等の
   定型句を出力する Whisper の既知問題が、ささやき声でまさに起きる

対策は「入力を大きく・きれいにする(上流)」と「誤りを吸収する(下流)」の両面で行う。

## 3. 改善要件

優先度: **P1 = 効果大・実装小(先にやる) / P2 = 実機計測とセットで判断 / P3 = P1-P2 で不足する場合**

### A. 入力段(録音品質)

- **R-A1 (P1) 自動ゲイン制御の有効化**
  `getUserMedia` の audio 制約に `autoGainControl: true` を追加する。
  受け入れ基準: iPhone 実機でささやき声の録音波形のピークが従来より大きいこと
  (VerifyScreen に波形 or RMS 表示を足して比較)。
- **R-A2 (P1) ソフトウェア音量正規化**
  `decodeToWhisperInput` 後の Float32Array を Whisper に渡す前にピーク正規化する
  (例: 最大振幅を 0.9 程度へスケール。無音判定済みの録音は対象外)。
  端末の AGC が効かない環境でも入力レベルを底上げできる、最も確実な一手。
  受け入れ基準: 正規化前後で同一録音の認識結果を VerifyScreen で比較し、悪化しないこと。
- **R-A3 (P1) 録音レベルメーターの表示**
  録音中に AnalyserNode で入力レベルをリアルタイム表示し、
  小さすぎるときは「もう少しマイクに近づけて」と促す。
  ユーザーが「声が届いていない」ことにその場で気づけるようにする(現状は結果が出るまで不明)。
- **R-A4 (P2) noiseSuppression / echoCancellation の A/B 検証**
  on/off を設定画面から切り替え可能にし、iPhone 実機でささやき声の認識率を比較する。
  ノイズ抑制がささやき声を削っているなら off を既定にする。結果は本書に追記。
- **R-A5 (P3) ハイパスフィルタ**
  OfflineAudioContext に BiquadFilter(highpass ~80Hz)を挿入し、
  ジムの低周波ノイズ(空調・振動)を除去する。R-A4 の計測で効果があれば採用。

### B. 前処理段(無音・幻覚対策 — design.md §6 の未実装分)

- **R-B1 (P1) RMS 音量ゲートの実装**
  録音全体の RMS を計算し、閾値未満なら推論に回さず「聞き取れませんでした。
  マイクに近づけてもう一度」を返す。閾値は実機のささやき声サンプルで較正する
  (正規化 R-A2 より前段で判定すること)。
- **R-B2 (P1) 幻覚ブラックリストの実装**
  「ご視聴ありがとうございました」「チャンネル登録」等の既知幻覚フレーズを
  パーサー前に除去し、全文が幻覚なら R-B1 と同じ再録音案内を出す。
  フレーズはコード内定数とし、実測で出たものをユニットテスト付きで追加していく。
- **R-B3 (P3) Silero VAD の前段挿入**
  ONNX でローカル実行できる無料 VAD で発話区間だけを切り出して Whisper に渡す。
  無音の混入が減り幻覚と処理時間の両方に効くが、モデル追加 DL とメモリ消費が
  増えるため、R-B1/B2 で不足する場合のみ。

### C. 推論段(モデル・デコード)

- **R-C1 (P2) モデル×設定の実機比較の完遂**(フェーズ1既存項目の再掲・具体化)
  whisper-base / whisper-small × (WASM / WebGPU) × (q4 / q8) を、
  **ささやき声の固定フレーズ集**(§5)で比較し、採用構成を決めて DEFAULT_CONFIG を更新する。
  kotoba-whisper v2.2 は 370MB 超のため除外済み(CLAUDE.md)。
- **R-C2 (P2) デコードオプションの実験**
  transformers.js の生成オプション(ビームサーチ `num_beams` 等)で精度が
  変わるかを VerifyScreen で試す。**transformers.js がどこまで対応しているかは未検証**のため、
  まず対応状況を確認し、効果と処理時間のトレードオフで判断する。
- **(非採用) ファインチューニング**
  ささやき声データでの追加学習は、データ収集・学習環境のコストが趣味開発の範囲を
  超えるため行わない。

### D. 後処理段(パーサーでの吸収)

- **R-D1 (P2) 清音・濁音・半濁音を考慮した編集距離**
  現在の levenshtein は全置換を等コストで扱う。ささやき声で混同しやすい
  ペア(ハ行⇔バ行⇔パ行、カ行⇔ガ行 等)の置換コストを下げた重み付き距離にし、
  「ペンチプレス」→「ベンチプレス」型の誤りを既存閾値のまま拾えるようにする。
  受け入れ基準: 混同ペアのテストケース追加+既存テスト全 PASS。
- **R-D2 (P1) 誤認識ログの蓄積導線**
  確認 UI で修正が発生したとき「認識テキスト → 確定した種目」のペアを
  IndexedDB に記録する。溜まったログから別名辞書(exercises.ts)と
  ユニットテストに反映する運用ループを作る(規約「誤認識パターンをテストケースとして蓄積」の実装)。
- **R-D3 (P3) 数値部分の混同補正**
  重量・回数の数字(イチ⇔シチ 等)にも混同対応を広げる。実測で数字の誤りが
  目立つ場合のみ。

### E. UX 段(最終手段の強化)

- **R-E1 (P2) ワンタップ再録音**
  認識失敗・低確信時に、結果画面から 1 タップで録り直せる導線を置く。
- **R-E2 (保留) 「ささやき声」要件自体の再検討**
  上記を実施しても実用精度(§5)に届かない場合、design.md §6 の方針どおり
  「小声(有声)を許容する」への要件緩和をユーザーと相談する。**勝手に緩和しない。**

## 4. 実施順序

1. **第1弾(P1)**: R-A1 → R-A2 → R-B1 → R-B2 → R-A3 → R-D2
   (すべて実装小。録音品質の底上げ+幻覚遮断+ログ収集開始)
2. **第2弾(P2)**: §5 の測定セットを整備し、R-A4 / R-C1 / R-C2 を実機計測。
   結果を見て R-D1 / R-E1 を実装
3. **第3弾(P3)**: 第2弾の結果が不足する場合のみ R-A5 / R-B3 / R-D3、
   それでも不足なら R-E2 の相談

## 5. 検証方法(受け入れ基準の共通枠)

- **測定セット**: 想定発話文法の固定フレーズ 20 件程度
  (例: 「ベンチプレス 60キロ 10回」…種目辞書から頻出種目+数値バリエーション)を、
  iPhone 実機・ジム相当の環境音下で**ささやき声**で録音して使い回す
- **指標**: パーサー確信度が `exact / alias` で通った率(=無修正で記録できた率)を主指標、
  `fuzzy` 込みの成功率と幻覚発生率を副指標とする
- **目標(提案値・ユーザーと要合意)**: 無修正成功率 80% 以上、幻覚が確認 UI に到達する率 0%
- 各要件の実装ごとに同一測定セットで前後比較し、結果を本書の末尾に追記する
